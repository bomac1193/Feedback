## The Veve-as-Topology Architecture

### Core Insight

**Veve is not a separate engine. It's the GRAPH that defines how engines couple.**

Currently, `kuramoto.genexpr` (lines 132-137) couples all-to-all — every oscillator pulls on every other:

```
sum1 = sin(t2-t1) + sin(t3-t1) + sin(t4-t1) + sin(t5-t1) + sin(t6-t1);
```

A veve graph makes coupling **selective**. Only connected nodes influence each other:

```
CURRENT (all-to-all):          VEVE (selective):
  1 ─ 2                         1 ─ 2
  │ × │    6 nodes,            │       only edges
  │ ╲ │    15 connections       │       in the veve
  3 ─ 4                        3 ─ 4    define coupling
```

### The Layered Architecture

```
P1 (Chaos)         = ENERGY   → excitation signal (what drives the system)
Veve Graph         = STRUCTURE → adjacency matrix (which connections exist)
P3 (Kuramoto)      = RHYTHM   → how energy self-organizes on the graph
P4 (Swarm)         = MELODY   → agents moving on the graph topology
TD Visual          = SAME GRAPH rendered as geometry + phase + diffusion

Switch veve live → rhythm reorganizes → visual reorganizes → sound transforms
```

### Why This Works

1. **In Max**: The adjacency matrix changes which coupling terms are active in the Kuramoto gen~. Different veve = different groove character. The modification is ~30 lines — change the hardcoded `sum1..sum6` to read from a Data buffer holding the adjacency matrix.

2. **In TD**: The same graph renders as node-edge geometry. Oscillator phases map to node color/pulse. Edge weights map to line thickness. Reaction-diffusion can flow along edges instead of a uniform grid.

3. **Both use the same data**: A veve is defined once (as a list of nodes + edges + weights). Max reads it for coupling. TD reads it for rendering. One source of truth.

### Key Topological Insight

The Legba veves are predominantly **radial/cross** — center hub connecting outward. This produces **leader-follower dynamics** in Kuramoto (one oscillator entrains the rest).

The Ogou veves are predominantly **linear/branching** — V-forks, chains, zigzags. This produces **wave propagation** in Kuramoto (sync travels along a path, creating ripple rhythms).

**Different lwa families = different groove physics.** This is not designed — it's discovered from the actual veve forms. The ancestral geometry contains rhythmic information.

### Digitization Process

1. Source veve drawings from Rigaud Ve-Ve plates (referenced above by PDF page number)
2. Trace in vector format (Inkscape/Illustrator) as planar graph
3. Identify nodes (intersections, endpoints, curve control points)
4. Identify edges (line segments between nodes)
5. Assign edge weights (line thickness, curvature → coupling strength)
6. Export as adjacency matrix + node XY positions (JSON format)
7. Max reads adjacency matrix → Kuramoto coupling via Data buffer
8. TD reads adjacency matrix + positions → visual rendering
9. Same source file, two modalities

### Scaling: Real Veves Have 15-30+ Nodes

The current Kuramoto runs 6 oscillators. Real veves have more structural complexity. Three options:

1. **Graph reduction** — Identify the 6 most topologically significant nodes (highest degree, most central). Lose detail but core structure is real.
2. **Scale up Kuramoto** — Run 16-24 oscillators. Computationally trivial in gen~. Richer rhythm, more faithful to veve.
3. **Hierarchical (recommended)** — Full veve graph in TD for visuals. Reduced major-node version in Max for audio. Both derived from same source. TD shows full cosmology, Max sounds the skeleton.

Option 3 is most practical. The visual shows the real veve. The audio sounds its major structural tensions.
